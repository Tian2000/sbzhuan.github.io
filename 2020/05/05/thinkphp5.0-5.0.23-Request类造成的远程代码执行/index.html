thinkphp5.0-5.0.23-Request类造成的远程代码执行

Hexo
2020-05-05

影响版本

ThinkPHP5.0.1-5.0.23完整版

漏洞分析

漏洞关键点在 thinkphp\library\think\Request.php 的第518行`

当 $method 的值为 false 和 $this->method 为空时，进入elseif块里面，

var_method的值默认存在且为 ‘_method’,所以我们可以通过post方法传递 _method参数, 这里thinkphp没有对该值进行验证，从而直接赋值给$this->method,然后调用Request类下的方法. 我们可以通过将 _method参数的值设为 ‘__construct’ 从而调用Request的构造方法。

thinkphp\library\think\Request.php的第135行

这里的$options参数通过前面的method方法处知道是$_POST数组，所以通过post参数可以对Request当前实例进行变量覆盖。

通过变量覆盖，可以覆盖Request的 filter属性、method属性以及get属性，

通过断点调试，可以看到第一次调用 think\Request->method()是从 App.php的643行路由检查时调用的

在获取了路由信息后在 App.php 139行执行路由调度

thinkphp\library\think\App.php 445行：

这里我们需要 $dispatch[‘type’] 为 ‘controller’ 或者 ‘method’ 的才能调用Request的param()方法，如何让条件达成后面再说

先进入param方法看看

thinkphp\library\think\Request.php 634行

进入$this->get(false)方法

进入了$this->input方法，这里的$this->get通过变量覆盖可控

因为$name 为false,所以 $this->input($this->get, $name, $default, $filter);返回了this->get.

再param方法的最后执行了 $this->input($this->param, $name, $default, $filter);

这是name为 空字符串不直接返回$this->param，向下执行到 $this->getFilter($filter, $default);

filter为空不为null,所以filter为$this->filter（可控）

回到input方法执行到 array_walk_recursive($data, [$this, ‘filterValue’], $filter);

查看 filterValue方法

call_user_func的$filter为$this->filter,$value为$this->get数组的值，都可控，所以造成任意代码执行

自动加载机制

THINKPHP5有自动类加载机制，会自动加载vendor目录下的一些文件。

vendor\topthink\think-captcha\src\helper.php

当通过url传参 ?s=captcha时，前面讲的$dispatch[‘type’] 为 ‘method’ ,然后就执行了 param()方法
